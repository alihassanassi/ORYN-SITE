Inside the ORYN Sensor

Published March 25, 2026 · 15 min read · By Ali Hassan Assi

Most managed security providers sell you a dashboard connected to someone else's detection engine. They pipe your logs to a cloud SIEM, run some correlation rules written three years ago, and email you an alert PDF once a month.

ORYN is different. It's a physical sensor that sits inside your network perimeter — between your firewall and your managed switch — and watches every packet that crosses the wire. No cloud. No third-party data access. No "we'll get back to you in 24 hours."

This post documents the production architecture: what it detects, how it detects it, and why the safety model matters as much as the detection model.

The Physical Layer: Firewall Behind the Firewall

ORYN deploys as a passive network tap. It connects to a mirror/SPAN port on your managed switch, receiving a copy of every packet that traverses your network. The sensor itself has no IP address on your production network. It cannot inject packets. It cannot modify traffic. It is architecturally incapable of disrupting your network.

This is a deliberate constraint, not a limitation. A security sensor that can take down your network is a liability. A sensor that can only observe is a tool. ORYN is a tool.

If the sensor hardware fails, traffic continues flowing through your switch unimpeded. We call this fail-closed safety with fail-open networking — the sensor shuts down safely, but your business never goes offline because of us.

Packet Capture and Protocol Analysis

At the lowest layer, ORYN captures packets and parses them into structured protocol events. The capture engine maintains a 10,000-packet circular buffer — enough for forensic context, small enough to prevent memory exhaustion. No raw payloads are stored to disk. Ever. Only metadata and evidence hashes.

The protocol analyzer handles five core protocols:

  • DNS: Every query and response is logged. NXDOMAIN rates are tracked per source IP — a sudden spike in failed DNS lookups is a classic indicator of domain generation algorithms (DGAs) used by malware for C2 communication.
  • HTTP/S: Method, status code, host header, and user-agent extraction. We don't decrypt TLS — but we don't need to. The metadata tells us what we need to know, and JA3 fingerprinting handles the rest.
  • SMB: Authentication events, share access, and failed login attempts. SMB is the primary vector for lateral movement in Windows networks, and most small businesses have flat networks where SMB traffic flows unrestricted.
  • RDP: Remote Desktop connection attempts. RDP brute force is one of the most common initial access vectors for ransomware operators targeting SMBs.
  • SSH: Banner extraction and authentication failure tracking. SSH brute force from external IPs triggers automatic alerting.

Eight Threat Detection Engines

Every detection engine maps to a MITRE ATT&CK technique. This isn't decorative — it means every alert comes with context about what class of attack is happening, not just "something looks weird."

1. Port Scan Detection (T1046 — Network Service Discovery)

If a single source IP touches more than 20 unique destination ports within 60 seconds, it's scanning. This catches both external attackers mapping your perimeter and internal hosts that have been compromised and are now mapping your internal network.

2. Brute Force Detection (T1110 — Brute Force)

More than 10 authentication failures from a single source in 60 seconds. Covers SSH, RDP, SMB, and HTTP Basic Auth. Rate-based detection with sliding windows, not static thresholds.

3. DNS Tunneling Detection (T1071.004 — Application Layer Protocol: DNS)

DNS tunneling encodes data in DNS queries to exfiltrate information or establish command-and-control channels. ORYN measures query entropy and payload size — legitimate DNS queries have low entropy and small payloads. Tunneling traffic has entropy above 3.5 bits/char and payloads exceeding 100 bytes.

4. C2 Beacon Detection (T1071 — Application Layer Protocol)

Command-and-control beacons call home at regular intervals with low jitter. ORYN tracks connection timing to external IPs and flags patterns where timing jitter drops below 5%. A human browsing the web has random, high-jitter connection patterns. Malware beacons are metronomic.

5. Lateral Movement Detection (T1021 — Remote Services)

Internal-to-internal connections on RDP (3389), SMB (445), or WMI (135) ports. In a flat network with no segmentation, every workstation can reach every server via these protocols. When one machine starts reaching out to others on these ports in rapid succession, something is spreading.

6. Data Exfiltration Detection (T1048 — Exfiltration Over Alternative Protocol)

If a single internal host sends more than 10 megabytes per minute to a single external IP, that's not normal web browsing. That's a potential data exfiltration event. ORYN tracks byte volumes per source-destination pair with sliding windows.

7. ARP Spoofing Detection (T1557.002 — ARP Cache Poisoning)

Gratuitous ARP packets and MAC/IP conflicts indicate someone (or something) is trying to intercept traffic via man-in-the-middle positioning. ORYN maintains a MAC-to-IP mapping table and alerts on conflicts.

8. Rogue DHCP Detection (T1557.003 — DHCP Spoofing)

Any DHCP server response from an IP not in the whitelisted DHCP server list generates an immediate alert. A rogue DHCP server can redirect all traffic through an attacker-controlled gateway.

JA3/JA3S TLS Fingerprinting

Every TLS connection starts with a handshake. The Client Hello message contains a unique combination of TLS version, cipher suites, extensions, elliptic curves, and point formats. The MD5 hash of this combination is the JA3 fingerprint. The Server Hello response produces a JA3S fingerprint.

Malware families have distinctive TLS fingerprints because their developers use specific TLS libraries and configurations. ORYN maintains a database of 50+ known-malicious fingerprints covering:

  • Cobalt Strike, Sliver, Mythic, Havoc, BruteRatel — commercial and open-source C2 frameworks
  • Emotet, Trickbot, QakBot, IcedID — banking trojans
  • AgentTesla, Remcos, AsyncRAT, NanoCore — remote access trojans
  • RedLine, Raccoon, Vidar, Mars — credential stealers

A JA3 match doesn't mean "maybe." It means a TLS handshake on your network exactly matches a known malware sample. That's a high-confidence detection.

Data Loss Prevention (DLP)

ORYN scans outbound traffic for sensitive data patterns:

  • Social Security numbers — XXX-XX-XXXX pattern with format validation
  • Credit card numbers — 13-19 digit sequences with Luhn checksum validation (not just regex — actual mathematical verification)
  • Bulk email addresses — 5+ email addresses in a single outbound payload indicates a potential contact list exfiltration
  • Credentials in cleartext — "password=", "passwd:", "Authorization: Basic" in unencrypted traffic
  • File signatures — ZIP, PDF, Office document, and database file magic bytes in outbound traffic

DLP only scans outbound traffic to external IPs. Internal traffic is excluded. And critically: ORYN never stores the matched data. Only evidence hashes and match counts are recorded. We prove a leak happened without keeping a copy of the leaked data.

Behavioral Baseline Profiling

Every IP on your network gets a rolling 24-hour behavioral profile tracking:

  • Byte and packet volumes (samples per minute)
  • Destination IP diversity
  • Protocol distribution
  • Active hours of day

Anomaly detection uses z-score analysis with a 2.5 sigma threshold. This means if an IP suddenly starts generating traffic that is 2.5 standard deviations above its normal behavior, ORYN flags it — even if the traffic doesn't match any known attack signature.

This is how you catch zero-days. Signature-based detection only catches known attacks. Behavioral baselines catch anything that deviates from normal — including attacks that haven't been discovered yet.

Four Intelligence Feeds

ORYN doesn't just watch your network — it watches the world and correlates:

  • NVD CVE Monitoring: Polls the National Vulnerability Database every 6 hours. Cross-references new CVEs against software observed on your network.
  • AbuseIPDB Reputation: Looks up every external IP your network communicates with against the AbuseIPDB threat database. Results are cached 24 hours to respect rate limits.
  • Certificate Transparency: Watches the Certificate Transparency logs for any certificates issued for your domains. Catches attackers who register lookalike domains or obtain fraudulent certificates.
  • Shodan Port Monitoring: Weekly checks of your external-facing IPs against Shodan. Detects new open ports, removed services, and configuration changes visible from the internet.

AI-Powered Event Correlation

The correlation engine is where individual events become a narrative. When ORYN sees the same source IP appear in three or more different event types within a 10-minute window, it automatically escalates severity to critical and routes the correlated alert to the human operator.

Example: an IP triggers a brute force alert, then appears in a lateral movement event, then shows up in an outbound data volume anomaly. Individually, each event might be medium severity. Together, they're a textbook attack chain: initial access → lateral movement → exfiltration.

The correlation engine sees this in real-time. A human reviewing individual alerts in a SIEM might not connect these events for hours or days.

Multi-Tenant Client Isolation

ORYN is built for MSSP (Managed Security Service Provider) deployment. Every event, baseline, IOC lookup, and certificate watch entry is tagged with a client identifier at the database layer.

Cross-client correlation is architecturally impossible — the database queries are scoped per client ID, not filtered after retrieval. This isn't a software policy. It's a structural guarantee. Client A's traffic patterns can never influence Client B's alerts.

The Safety Model

ORYN's safety architecture is built on three principles:

  1. Passive only. Zero packet injection. The sensor physically cannot send traffic onto your network. This is enforced by the network tap configuration, not by software.
  2. Fail-closed sensor, fail-open network. If the sensor crashes, your traffic continues flowing. If the sensor can't make a determination, it drops the event rather than guessing. If the kill switch file exists, all detection engines stop immediately.
  3. No raw data storage. Packet payloads are analyzed in memory and discarded. Only metadata, evidence hashes, and structured events are persisted. You can prove an event happened without storing the actual content.

A hardware sentinel file provides a bypass mechanism. If the file is removed (indicating hardware failure or operator intervention), the sensor enters safe mode and stops all processing. Your network is never held hostage by our sensor.
