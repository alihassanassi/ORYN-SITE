The Constitution Before the Product

Published March 25, 2026 · 14 min read · By Ali Hassan Assi

Every AI company talks about safety. Most of them mean content filters. Some of them mean RLHF. Almost none of them mean architecture.

This paper documents how JARVIS — a locally-run AI assistant built for cybersecurity research — enforces safety constraints through structural design rather than behavioral training. The system runs entirely on local hardware, uses no cloud inference, and treats every autonomous action as potentially dangerous until proven otherwise.

The thesis is simple: if your AI can be talked out of its safety rules, they aren't safety rules. They're suggestions.

The Problem with Policy-Based Safety

Most AI safety today operates at the prompt level. A system prompt says "don't do harmful things." Guardrails check outputs against blocklists. RLHF shapes the model to refuse certain requests.

This works until it doesn't. Jailbreaks exist because policy-based safety is fundamentally a content filtering problem masquerading as an engineering problem. You can always find a way to phrase something that slips through a content filter. You can't engineer your way past a gate that requires a cryptographic token to open.

The Five-Layer Architecture

JARVIS is built on a strict 5-layer stack. Each layer only depends on layers below it. No layer can reach up to modify a layer above it.

  • Layer 1 — Configuration: Static config, security policies, secrets management. Nothing at runtime can modify this layer.
  • Layer 2 — State & Memory: Database, audit logs, memory systems. All state changes are recorded with tamper detection.
  • Layer 3 — Execution: Agents, tools, voice pipeline, LLM interface. This is where work happens — but work is bounded.
  • Layer 4 — Interface: GUI, voice I/O, operator-facing systems. Presentation only — no execution authority.
  • Layer 5 — Autonomy: Scheduler, overnight learner, strategy engine. The most powerful layer — and the most constrained.

The critical rule: Layer 3 never imports Layer 5. Execution cannot invoke autonomy. Autonomy can request execution, but execution doesn't know autonomy exists. This prevents any tool or agent from escalating its own privileges.

The Seven-Gate Safety Chain

Every autonomous action — every scan, every report, every external request — passes through a serial chain of seven gates. All gates are fail-closed: if any gate cannot make a determination, the action is denied.

  1. Gate 1 — Authorization: Is the kill switch active? Is autonomy enabled? Has the operator approved this class of action?
  2. Gate 2 — Scope Enforcement: Is this action within the declared target scope? Any action targeting assets outside scope is rejected unconditionally.
  3. Gate 3 — Rate Limiting: Has this action type exceeded its budget? Every action class has a configurable rate limit with a cooldown period.
  4. Gate 4 — Safety Classification: What is the risk level of this action? High-risk actions require explicit operator confirmation.
  5. Gate 5 — Deconfliction: Is another action currently running that would conflict? No parallel operations on the same target.
  6. Gate 6 — Evidence Quality: Is there sufficient evidence to justify this action? Low-confidence findings are flagged, never auto-actioned.
  7. Gate 7 — Audit Recording: Before execution, the action is written to a hash-chained audit log. If the log write fails, the action does not proceed.

The chain is serial. There is no "skip to gate 7." There is no "gate 2 was already checked." Every action starts at gate 1, every time.

The Kill Switch

JARVIS has a multi-layer kill switch that cannot be overridden by any software component:

  • Filesystem sentinel: A file called EMERGENCY_STOP.flag. If this file exists, all autonomous operations halt. Any process can create it. No process can delete it except the operator.
  • Keyboard shortcut: Ctrl+Alt+Shift+K — immediately creates the sentinel and halts all autonomous loops.
  • Python state flag: An in-memory flag that is checked before every autonomous cycle. Redundant with the filesystem sentinel.

The kill switch is checked at the top of every autonomous loop, every gate evaluation, and every tool dispatch. It is not a graceful shutdown — it is an immediate halt.

Prompt Injection Defense

Because JARVIS processes untrusted text from web sources during research, prompt injection is a real threat. The defense is structural:

  • All untrusted text is wrapped in delimiters that the LLM is instructed to treat as data, not instructions.
  • The security sanitizer strips known injection patterns before text reaches the LLM.
  • Research synthesis operates in a constrained JSON extraction mode — the model outputs structured data, not freeform responses.
  • No synthesized content can trigger tool execution. The bridge between research and action requires operator approval.

What This Means

JARVIS is not a safe AI because it was trained to be safe. It's a safe AI because it was built to be constrained. The difference matters.

A model that was trained to refuse harmful requests can be retrained, fine-tuned, or jailbroken. A system that requires a filesystem sentinel to be absent before it can take any autonomous action cannot be talked out of checking for that file.

This is not a novel insight. It's how we build every other safety-critical system. Nuclear reactors don't rely on the fuel rods' "preference" not to melt down. They have physical control rods that absorb neutrons regardless of what the reaction "wants." JARVIS applies the same principle to AI: the constraints are structural, not behavioral.

If you can convince your AI to ignore its safety rules, you don't have safety rules. You have a suggestion box.
